What We Are Actually Optimizing

The architecture is shaped by three non-negotiables: tactical correctness, coaching usability, and operational reliability.

Tactical Correctness

Outputs must reflect game model, phase, zone, and role responsibilities without tactical contradictions.

Example failure to avoid: high-press objective + low block drill behavior

Coaching Usability

Instructions must be teachable in a live session context, with clear setup, cues, and progression.

Coach-level adaptation: Grassroots -> simple language USSF C/B+ -> advanced tactical cues

Reliability

System should degrade gracefully on auth/network/model issues and still preserve user momentum.

Policy: retry -> recover -> fallback never silently fail
Tradeoff: tighter guardrails reduce some creative variance, but substantially improve trust and repeatability for coaches.

Prompt System

Prompt and Constraint Design: Why It Matters

Prompting is treated as architecture, not copywriting. Each parameter influences tactical framing and coach-facing language.

Input Mapping Layer

  • gameModel/phase/zone -> tactical intent and scenario conditions.
  • ageGroup/playerLevel -> cognitive and physical complexity envelope.
  • coachLevel -> vocabulary and instruction granularity.
  • duration/counts -> realistic drill budget and setup scope.

Output Contract Layer

  • Required fields: organization, coaching points, progressions, constraints.
  • Time-distribution sanity checks across session phases.
  • Diagram compatibility with renderer expectations.
  • Load notes and flow coherence checks for drill order.
prompt = template + context(gameModel, phase, zone) + pedagogy(ageGroup, playerLevel, coachLevel) + contract(required_json_fields) + anti-patterns(to avoid tactical contradictions)
Quality Engine

Quality Framework and Gating Strategy

Quality is enforced through explicit dimensions, thresholds, and a deterministic pass/fail policy.

Scoring Dimensions

  • Tactical coherence with selected objective.
  • Progression quality from drill to drill.
  • Age/level appropriateness and safety envelope.
  • Coach usability in live-session execution.
score_total = w1*tactical + w2*progression + w3*age_fit + w4*usability

Gate Policy

  • Hard schema failures -> immediate fix path.
  • Low tactical/progression scores -> targeted fixer.
  • Persistent failure -> controlled, transparent error.
  • No silent acceptance of weak outputs.
Design decision: predictable floor quality is prioritized over occasional high-variance creativity.
Fixer + Failure Modes
Page 7 / 10

Auto-Fix Loop and Failure Containment

The fixer system localizes corrections so good content is retained while failing dimensions are repaired.

Common Failure Modes

  • Weak progression narrative across drills.
  • Coach cues too generic for chosen level.
  • Time allocation imbalance in session plan.

Fixer Strategy

  • Use QA reasons as patch instructions.
  • Rewrite only failing sections.
  • Preserve validated tactical core.

Containment Outcome

  • Lower user-facing failure rate.
  • Higher consistency in final outputs.
  • Clear logs for post-incident analysis.
if fail_reasons: patched = fixer(original, fail_reasons) validated = validate(patched) return validated if pass else controlled_error

How Progressive Series Works (Included and Expanded)

A progressive series is generated as a curriculum arc with continuity constraints, not as isolated session calls.

Series Timeline Logic

1
Establish Base Pattern

Build shared tactical language and core mechanics.

2
Add Constraints

Increase decision speed and environmental pressure.

3
Increase Complexity

Introduce more dynamic tactical interactions.

4
Transfer to Game

Test objective under near-match conditions.

Series Integrity Checks

  • Progression index must trend upward by design.
  • Duplication control across drills and language cues.
  • Per-session QA still required before inclusion.
  • Objective alignment checked across entire series.
series_pass = all(session.qa_pass) && progression_delta >= min && duplication <= max && objective_alignment >= min

Persistence, Traceability, and Operational Reliability

Durable artifacts and resilient request handling are key to repeatable coach experience.

Vault Data Model Value

  • Session/series saved with references and metadata.
  • Enables lookup by ref code and audit trails.
  • Supports favorites, scheduling, and player-plan flows.
  • Creates continuity between generation and execution.

Reliability Patterns

  • API fallback routing for local/dev connectivity variance.
  • Token refresh + anonymous retry for recoverable auth errors.
  • Stable typed responses reduce frontend fragility.
  • Structured logs improve MTTR on generation incidents.
on_invalid_token: try_refresh_once() if refresh_fail: clear_auth_state() retry_request(anonymous_if_allowed)
